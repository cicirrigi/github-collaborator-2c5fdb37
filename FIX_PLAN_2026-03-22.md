# Vantage Lane 2.0 Fix Plan

Date: 2026-03-22
Based on: `CODEBASE_AUDIT_2026-03-22.md`

## Goal

Bring the repository from "active transition state" to a stable, coherent, releasable baseline.

This plan is ordered to reduce risk first, then restore correctness, then improve maintainability.

## Working Principles

1. Do not patch symptoms before choosing the canonical architecture.
2. Restore one green baseline before adding more features.
3. Remove parallel code paths where possible instead of maintaining both.
4. Treat booking write paths and payment state transitions as business-critical.

## Phase 0: Freeze and Clarify

Objective:

- stop architecture drift from getting worse while fixes are in progress

Actions:

1. Freeze new feature work in booking, billing, and payment until P0 is done.
2. Decide which booking flow is canonical:
   - current API + `buildBookingPayload` / `buildLegsPayload`
   - or a newer atomic RPC-first flow
3. Decide which Step 1 UI is canonical:
   - `BookingFormCard`
   - or `PremiumBookingFormCard`
4. Decide whether payment RPC mode is now the standard and old direct DB mode should be retired.

Done when:

- one canonical path is chosen for booking creation
- one canonical path is chosen for payment processing
- one canonical Step 1 UI path is chosen

## Phase 1: Fix Business-Critical Data Integrity

Objective:

- prevent inconsistent booking data from being written

### 1. Fleet booking consistency

Priority: P0

Tasks:

1. Audit intended data model for fleet:
   - one leg per vehicle instance
   - one leg per route only
   - one job per vehicle request
2. Align `buildLegsPayload()` with that model.
3. Align `/api/bookings` fleet job creation with the same model.
4. Add at least one deterministic test or fixture-based validation for:
   - 1 vehicle x quantity 1
   - 1 vehicle x quantity > 1
   - multiple vehicle categories with quantities

Files likely involved:

- `src/services/booking-mapping/dbPayload.ts`
- `src/app/api/bookings/route.ts`
- SQL tables / RPC assumptions around `booking_legs`, `booking_jobs`, `booking_vehicle_requests`

Done when:

- fleet bookings produce internally consistent legs, requests, and jobs
- expected relationships are documented

### 2. Booking + quote atomicity

Priority: P0

Tasks:

1. Compare active route logic against `create_booking_with_quote_atomic`.
2. Decide whether quote creation is mandatory for all bookings.
3. If yes, migrate active booking creation to the atomic RPC.
4. If not yet, document clearly why current split write path remains acceptable.

Files likely involved:

- `src/app/api/bookings/route.ts`
- `sql/migrations/20260318_create_booking_with_quote_atomic.sql`

Done when:

- booking write path matches intended transactional guarantees
